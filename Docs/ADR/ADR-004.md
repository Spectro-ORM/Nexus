# ADR-004: Throws vs Halting — Error Signalling Contract

**Status:** Accepted
**Date:** 2026-03-25
**Owner:** Alembic Labs

---

## Context

A plug pipeline can fail in two qualitatively different ways:

1. **HTTP-level rejection** — The request is syntactically valid, but the server wants to
   respond with a non-success status (401 Unauthorized, 403 Forbidden, 404 Not Found,
   422 Unprocessable Entity, etc.). The pipeline should stop, but the server should still
   send a well-formed HTTP response to the client.

2. **Infrastructure failure** — A database is unreachable, an I/O operation timed out, a
   dependency panicked. These are unexpected conditions that the outer server layer must
   catch and convert into a 500 response (or similar) with appropriate logging.

If plugs signal both failure kinds by throwing, distinguishing them requires error type
inspection at every call site. If everything halts, infrastructure failures become
invisible to callers.

## Decision

The contract for plugs is:

| Scenario | Mechanism |
|---|---|
| HTTP-level rejection (4xx, deliberate 5xx) | Set `response.status`, populate `responseBody`, and return `connection.halted()`. |
| Infrastructure failure | `throw` an `Error`. |

**Halting** terminates the plug pipeline without an exception. The server layer delivers
the response already assembled in the `Connection` to the client.

**Throwing** propagates out of the pipeline. The server adapter (e.g. `NexusHummingbird`)
catches these, logs them, and responds with a generic 500.

Plugs MUST NOT `throw` to signal intentional HTTP error responses. Doing so bypasses the
response-assembly pipeline (e.g. error-formatting plugs, logging plugs) that run after
the failing plug when using halt-based composition.

## Consequences

**Positive**
- Clear, auditable contract: inspecting a plug tells you exactly which branch is HTTP
  logic and which branch is infrastructure recovery.
- Error-shaping plugs (JSON error formatter, HTML error page renderer) can be inserted
  at the top of the pipeline and will always see halted connections — even for auth
  failures deep in the stack.
- Infrastructure errors propagate without accidental swallowing.

**Negative / Trade-offs**
- Plug authors must learn the convention — throwing an `HTTPError` is tempting but wrong.
  Documented in `CONTRIBUTING.md` and enforced via code review.
- There is no compile-time enforcement of the "don't throw for HTTP errors" rule; it is a
  convention, not a type constraint.
