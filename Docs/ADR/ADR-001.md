# ADR-001: Use swift-http-types Rather Than Own HTTP Primitives

**Status:** Accepted
**Date:** 2026-03-25
**Owner:** Alembic Labs

---

## Context

Nexus needs types to represent HTTP methods, status codes, header fields, request lines,
and response lines. We could define our own enumerations and structs, but HTTP is a
well-specified standard with a large surface area — including hundreds of header field
names, dozens of status codes, and nuances around case-sensitivity and encoding.

Apple maintains [`swift-http-types`](https://github.com/apple/swift-http-types), a
purpose-built, pure-Swift package for exactly these primitives. It is:

- Designed for Swift 6 strict concurrency (`Sendable` throughout).
- Used by Apple's own networking stack (including Hummingbird 2 and Swift OpenAPI).
- Actively maintained and versioned.
- Zero-dependency itself.

## Decision

The `Nexus` core target depends on `swift-http-types` for all HTTP primitive types
(`HTTPRequest`, `HTTPResponse`, `HTTPField`, `HTTPFields`, `HTTPRequest.Method`,
`HTTPResponse.Status`, etc.).

We do **not** define our own equivalents of these types. Any Nexus-specific extensions
(convenience helpers, semantic wrappers) are added in `extension` blocks rather than
shadow types.

## Consequences

**Positive**
- No reinvention of well-tested, heavily used HTTP primitives.
- Interoperability with Hummingbird, Swift OpenAPI, and any other package that adopts
  `swift-http-types` — connections can flow across library boundaries without conversion.
- Free compliance with Swift 6 strict concurrency requirements.

**Negative / Trade-offs**
- Nexus carries one external dependency in its core target. This is acceptable because
  `swift-http-types` has no transitive dependencies of its own.
- API shape is dictated by Apple; Nexus cannot control breaking changes. Mitigated by
  pinning to a major version range.
